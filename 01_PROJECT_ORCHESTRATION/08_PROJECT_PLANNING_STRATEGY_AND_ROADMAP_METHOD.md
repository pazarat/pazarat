# PROJECT PLANNING STRATEGY AND ROADMAP METHOD

# Purpose

This file defines how the model should help plan projects, structure strategy, define roadmap phases, prioritize work, sequence artifacts, identify dependencies, and guide execution without confusing roadmap with folder structure or project identity.

It is part of the project orchestration layer.

It is not a project instance file.

It is not a project-specific roadmap.

It is not a project status report.

It is not a fixed execution plan for any current project.

It must not store fixed claims about any current project, artifact, file, folder, maturity, weakness, strength, priority, or roadmap.

Its purpose is to define a general method for planning any project or structured body of work.

The project orchestration layer defines how to plan.

The project instance layer stores the actual project-specific roadmap, priorities, decisions, phases, and execution plan.

---

# Core Principle

Planning is sequencing toward outcomes.

A roadmap is not a folder tree.

A roadmap is not a list of every possible file.

A roadmap is not proof of maturity.

A good roadmap connects:

- identity
- objectives
- scope
- knowledge needs
- current maturity
- dependencies
- risks
- artifacts
- decisions
- execution constraints
- next useful action

The model must not jump to implementation before foundations are ready.

The model must not keep discussing foundations when the project is ready for execution.

---

# What This File Owns

This file owns the project-orchestration method for planning, strategy, prioritization, and roadmap design.

It defines:

- how to decide what should happen next
- how to distinguish strategy from roadmap
- how to distinguish roadmap from structure
- how to sequence project work
- how to prioritize artifacts
- how to identify dependencies
- how to detect blockers
- how to handle uncertainty in planning
- how to create progressive project stages
- how to avoid over-planning
- how to avoid under-planning
- how to convert accepted decisions into roadmap items
- how to route project-specific roadmaps into the project instance layer

This file does not own:

- the current roadmap of a specific project
- current priority of a specific module
- current delivery schedule of a specific project
- project-specific implementation tasks
- project-specific release plans
- project-specific milestones
- project-specific resource allocation
- project-specific sprint planning

Those belong in the project instance layer or current project response when explicitly assessed.

---

# Strategy Definition

Strategy explains how the project should achieve its objectives.

It answers:

- what matters most
- what should be protected
- what must be avoided
- what sequence reduces risk
- what principles guide tradeoffs
- what should be built or understood first
- what should be deferred
- how the project should grow

Strategy is broader than a task list.

Strategy gives direction to the roadmap.

---

# Roadmap Definition

A roadmap is an ordered plan of work.

It answers:

- what comes first
- what comes next
- what can wait
- what depends on what
- what artifacts or decisions are needed before others
- what stage the project is in
- what is the next useful milestone
- what outputs should exist at each phase

A roadmap should be flexible enough to evolve.

It should be specific enough to guide action.

---

# Roadmap Versus Structure

Do not confuse roadmap with structure.

## Structure

Answers where things belong.

Examples:

- standards folder
- knowledge folder
- modules folder
- research folder
- implementation notes folder

## Roadmap

Answers what to do and when.

Examples:

- stabilize identity first
- create knowledge map before final artifact
- write parent scenario before child scenario
- research compliance before implementation
- generate project instance standards before deep project content

A folder may exist before it is ready to be worked on.

A roadmap item may exist before its folder exists.

---

# Roadmap Versus Artifact List

A roadmap is not merely a list of files.

A file list says what may exist.

A roadmap says why, when, and in what order work should happen.

A file should appear in the roadmap only when it contributes to a stage, decision, outcome, or dependency.

Do not create roadmap noise by listing every possible artifact.

---

# Planning Inputs

Before proposing a roadmap, consider:

- project classification
- project identity
- project objectives
- current scope
- current project instance state
- existing standards
- existing artifacts
- maturity of current artifacts
- knowledge gaps
- dependencies
- risks
- user’s current intent
- accepted decisions
- open questions
- desired output
- available time or resources if known

If inputs are incomplete, plan provisionally.

Do not pretend the plan is final.

---

# Planning Source Priority

When planning for a specific project, use:

1. current project instance standards
2. project identity and objectives
3. current project structure and artifacts
4. current conversation and user decisions
5. maturity assessment
6. knowledge gap assessment
7. project orchestration method
8. general knowledge

Do not create a project-specific roadmap from general methodology alone when project context exists.

---

# Planning Without Project Instance

If no project instance exists, the model can still help plan.

It may provide:

- lightweight next step
- concept roadmap
- discovery plan
- bootstrap sequence
- questions to clarify
- possible project stages
- recommendation to create a project instance if continuity is needed

Do not claim the roadmap is project-instance verified.

Treat it as proposed.

---

# Planning With Project Instance

If a project instance exists, planning should respect it.

The model should:

- account for the project-specific numbered tunnel
- use project identity and objectives
- inspect relevant artifacts when possible
- identify current maturity
- identify missing standards
- identify dependencies
- recommend next action based on project evidence
- avoid generic roadmap templates

Project-specific roadmaps belong in the project instance layer.

---

# Progressive Planning Stages

A project may move through these general stages.

## Stage 0 — Exploration

The idea is raw.

Goal:

- understand intent
- identify work type
- avoid premature structure

Useful outputs:

- organized summary
- concept map
- one focused question

## Stage 1 — Bootstrap

The work is becoming a project candidate.

Goal:

- stabilize identity
- clarify objectives
- decide whether a project instance is needed

Useful outputs:

- provisional identity
- objectives draft
- standard seed
- project instance recommendation

## Stage 2 — Standard Layer Formation

The project needs durable memory.

Goal:

- create project-specific standards
- define structure
- capture decisions
- identify knowledge needs

Useful outputs:

- identity and objectives
- structure index
- knowledge map
- decision log
- reinforcement file

## Stage 3 — Core Architecture Or Concept Development

The project needs its conceptual backbone.

Goal:

- define domains
- define major areas
- define relationships
- identify core artifacts

Useful outputs:

- domain taxonomy
- scope model
- parent artifacts
- research framework
- curriculum map
- system model

## Stage 4 — Artifact Production

The project is ready for detailed outputs.

Goal:

- produce PRDs, workflows, research notes, lessons, policies, implementation notes, or other artifacts

Useful outputs depend on project type.

## Stage 5 — Execution Or Delivery Planning

The project is ready for implementation, publication, study execution, research synthesis, operational rollout, or delivery.

Goal:

- define tasks
- manage dependencies
- handle risks
- align outputs with standards

Useful outputs:

- delivery roadmap
- implementation plan
- review plan
- release plan
- publication plan
- study plan

## Stage 6 — Evolution And Governance

The project is ongoing.

Goal:

- preserve decisions
- update standards
- manage changes
- prevent drift
- audit maturity

Useful outputs:

- change log
- decision updates
- standard revisions
- project audits
- roadmap updates

These stages are not rigid.

Use the project context.

---

# Minimum Useful Next Step

The model should often recommend the smallest useful next step.

A good next step should:

- reduce uncertainty
- preserve important context
- unlock future work
- avoid unnecessary complexity
- match the user’s current intent
- respect project maturity
- be easy to execute

The next step may be:

- answer a focused question
- create a seed file
- extract identity
- create structure map
- create knowledge map
- generate one artifact
- audit a folder
- resolve a conflict
- update a standard
- defer a premature task

Avoid recommending a large roadmap when one clear next file is needed.

---

# Prioritization Criteria

When choosing priorities, consider:

## Dependency

What must exist before later work?

## Risk Reduction

What prevents major future mistakes?

## Context Preservation

What prevents repeated explanation?

## User Value

What helps the user most now?

## Project Truth

What stabilizes the project’s core meaning?

## Knowledge Need

What must be researched or understood first?

## Execution Readiness

What is needed before implementation or delivery?

## Review Burden

What reduces future review fatigue?

## Reversibility

What can be changed later with low cost?

## Momentum

What moves the project forward safely?

Prioritization should be reasoned, not automatic.

---

# Dependency Detection

A roadmap should identify dependencies.

Examples:

- project identity before project structure
- knowledge map before high-stakes artifact
- parent artifact before child artifact
- standard layer before repeated artifact generation
- source policy before research synthesis
- workflow model before screen specification
- decision log before major architecture changes
- glossary before terminology-heavy documentation
- scope before roadmap expansion

Dependencies vary by project type.

Do not force product dependencies onto research or learning projects.

---

# Blocker Detection

A blocker is something that prevents reliable progress.

Possible blockers:

- unclear identity
- conflicting objectives
- missing project instance standards
- unresolved project type
- missing domain knowledge
- outdated artifact
- conflicting files
- wrong layer ownership
- unstable terminology
- missing parent context
- missing source policy
- unavailable repository
- broken output format
- excessive ambiguity

When a blocker exists, recommend resolving it before deeper work.

Do not overstate minor uncertainties as blockers.

---

# Risk-Aware Planning

Planning should detect risk.

Risk may include:

- building on weak assumptions
- documenting wrong project identity
- creating too many empty files
- skipping knowledge research
- implementing before standards
- overfitting general method to one project
- weakening a mature project during migration
- losing user corrections
- creating artifacts that future AI cannot navigate
- making current facts permanent without update policy

Roadmaps should reduce risk progressively.

# Roadmap File Placement

A project-specific roadmap belongs in the project instance layer.

Possible path:

    02_MY_PROJECT/08_PROJECT_DELIVERY_ROADMAP.md

or another project-specific equivalent.

If the project has a complex delivery structure, roadmap content may also live in a non-numbered project content folder after the standard tunnel.

Do not store a current project-specific roadmap inside the general orchestration layer.

---

# Roadmap Content

A project-specific roadmap may include:

- current project stage
- accepted objectives
- active priorities
- dependencies
- blockers
- phases
- milestones
- artifacts to produce
- knowledge work needed
- decision points
- risks
- deferred items
- next actions
- update policy

Exact content depends on project type.

---

# Roadmap Maturity

A roadmap may be:

## Provisional

Based on incomplete information.

## Basic

Identifies stages and next steps.

## Developed

Includes dependencies, risks, and artifacts.

## Execution-Ready

Supports implementation, publication, study, or delivery.

## Governed

Maintained as project decisions evolve.

Do not call a roadmap mature because it is long.

Assess whether it guides action.

---

# Avoid Over-Planning

Over-planning occurs when the model creates heavy plans before the project needs them.

Signals:

- too many phases too early
- too many files proposed
- roadmap not tied to objectives
- planning replaces progress
- user wanted one file but received a strategy essay
- speculative future work dominates current task

Avoid over-planning by focusing on the next useful action.

---

# Avoid Under-Planning

Under-planning occurs when the model jumps into outputs without foundations.

Signals:

- generating detailed artifacts before identity
- creating child artifacts before parent context
- designing implementation before domain clarity
- ignoring knowledge gaps
- skipping project standards
- treating raw ideas as final requirements
- no roadmap for complex work

Avoid under-planning by identifying dependencies and blockers.

---

# Planning During Migration

When the repository architecture is being migrated:

- preserve useful existing rules
- identify old-to-new layer mapping
- avoid regenerating everything blindly
- prioritize foundational layers first
- test with a real project instance
- fix output safety issues as they appear
- avoid weakening existing project-specific work
- produce files in a clear sequence

Migration itself needs a roadmap.

But do not make the roadmap more complex than the migration.

---

# Planning And Project Instances

A large active project can serve as a practical validation case.

The roadmap method should help decide:

- which project-local standards are already strong
- which project instance files are needed
- which artifacts should be written next
- which child artifacts depend on parent maturity
- what knowledge gaps affect quality
- where implementation can begin later

However, the active project’s specific roadmap belongs in the active project instance, not in this general orchestration file.

---

# Planning Output Patterns

Use different output patterns depending on intent.

## Quick Next Step

Use when the user asks what to do next.

Include one clear recommendation and why.

## Phase Map

Use when the project needs sequencing.

Include stages and main outputs.

## Roadmap Proposal

Use when the user wants a plan.

Include phases, dependencies, and next action.

## Blocker Report

Use when progress is unsafe without resolving an issue.

Include blocker, impact, and resolution.

## Project Roadmap File

Use when the user asks to generate a roadmap artifact.

Place it in the project instance layer.

---

# Decision Points

A roadmap should identify decisions that affect sequence.

Examples:

- whether to create project instance now
- whether identity is accepted
- whether project type is product, research, learning, or knowledge base
- whether knowledge research is required before artifact generation
- whether to generate parent artifact before children
- whether to migrate legacy folders
- whether to preserve old project structure or restructure

Decision points should not be hidden inside task lists.

---

# Deferred Items

Not every valuable idea belongs in the current phase.

Deferred items should be preserved when useful.

A deferred item may be:

- future artifact
- future project branch
- future feature
- future research topic
- future standard
- future implementation plan
- future migration step

Deferral is not deletion.

It protects current focus.

---

# Roadmap And Decision Memory

Important roadmap decisions should be recorded in the project instance decision memory when the project supports it.

Examples:

- accepted file sequence
- accepted project stages
- accepted deferrals
- accepted project type
- accepted migration plan
- accepted dependency order
- accepted priority

Do not rely only on conversation if the decision should persist.

---

# Roadmap And Knowledge

Knowledge work may be a roadmap stage.

Examples:

- research before writing
- source policy before synthesis
- glossary before documentation
- domain model before workflow
- compliance review before implementation

Do not treat knowledge work as secondary when it controls quality.

---

# Roadmap And Artifact Generation

Artifact generation should follow roadmap logic.

Before generating an artifact, check:

- is this artifact needed now?
- does it depend on another artifact?
- is parent context mature enough?
- is knowledge sufficient?
- is the user asking for final or provisional output?
- should this artifact be in project instance?
- will this artifact reduce future work?

Do not generate artifacts only because they are next numerically.

Numeric order controls tunnel reading, not always execution priority.

---

# Roadmap And Active Sequence

If an active file-generation sequence exists, the roadmap should respect it.

When the user says "التالي", continue the sequence unless a serious blocker appears.

Do not interrupt every file generation with new roadmap analysis.

If the sequence itself becomes wrong, explain briefly and correct it.

---

# Minimal Question Rule

Ask one focused planning question only if needed.

Useful questions:

- Is this intended to become a persistent project instance?
- Should we stabilize identity before generating artifacts?
- Is the priority now structure, knowledge, or execution?
- Should this roadmap target a product, research, learning, or knowledge base?
- Is this phase accepted or still exploratory?

Do not ask if the next step is already clear.

---

# Roadmap Update Method

Roadmaps should evolve when:

- identity changes
- objectives change
- project type changes
- standards are added
- knowledge gaps are resolved
- blockers appear
- artifacts mature
- user changes priority
- repository structure changes
- implementation begins

Update roadmaps deliberately.

Do not silently drift.

---

# Roadmap Audit

When auditing a project roadmap, check:

- alignment with identity
- alignment with objectives
- current project stage
- dependency order
- knowledge needs
- risk handling
- artifact sequence
- decision points
- deferred items
- next action clarity
- consistency with project standards
- whether the roadmap is too heavy or too vague

Roadmap audit findings belong in the current response or project instance audit artifact, not in this general file.

---

# Failure Signals

Planning failures include:

- confusing roadmap with folder tree
- confusing roadmap with file list
- building too much too early
- generating artifacts before foundations
- planning without project type
- ignoring knowledge gaps
- ignoring dependencies
- ignoring active sequence
- asking too many planning questions
- making static roadmap claims in general files
- storing one project’s roadmap in project orchestration
- weakening practical execution through excessive abstraction
- continuing discussion when the user asked for the next file

When failure occurs, correct the plan and continue.

---

# Final Principle

Planning should turn project understanding into ordered progress.

The model must choose the next action by considering identity, objectives, scope, maturity, knowledge, dependencies, risk, and user intent.

A strong roadmap helps the user move forward without losing meaning or building on weak foundations.
