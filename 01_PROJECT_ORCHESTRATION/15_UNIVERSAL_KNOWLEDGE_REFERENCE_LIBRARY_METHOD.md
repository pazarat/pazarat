# UNIVERSAL KNOWLEDGE REFERENCE LIBRARY METHOD

# Purpose

This file defines how the model should use a universal knowledge reference library when building, discussing, analyzing, planning, designing, documenting, or implementing projects across many domains.

It is part of the project orchestration layer.

It is not a project instance file.

It is not a domain encyclopedia.

It is not a static dump of every professional standard.

It is not a replacement for project-specific truth, accepted project decisions, local taxonomies, local UI standards, implementation standards, source policies, or project artifacts.

Its purpose is to establish a durable gateway that teaches the model how to route raw user intent through appropriate disciplinary alphabets, professional standards, knowledge bodies, quality principles, and evidence expectations before producing project work.

The project orchestration layer owns the method.

The project instance layer owns the project-specific application of the method.

---


# Project-Neutral Library Rule

The universal knowledge reference library is not built for one project instance.

It must support any project or work type that needs domain-aware expert help, including software products, research projects, scientific documentation, academic work, courses, operations, policy, design, engineering, business systems, or mixed-domain projects.

Project-specific examples, names, paths, scenarios, and accepted decisions belong in the project instance layer.

This orchestration file may define how to activate, expand, verify, and combine domain alphabets, but it must not make any one active project the default example or universal target.

If a project instance needs a specialized application of this method, that application must be documented in the project-local tunnel or artifact structure, not in the universal library method.

---

# Core Principle

Every serious project is built from one or more knowledge domains.

Each domain has an alphabet.

An alphabet is the set of stable concepts, rules, quality expectations, methods, standards, constraints, vocabulary, and reasoning patterns that define competent work in that domain.

The model must not treat a raw user idea as enough to build a mature project artifact.

The model must infer the involved domains, activate the relevant domain alphabets, adapt them to the user intent, and then generate project work that carries a consistent level of professional maturity across all involved domains.

The model must use the library as a gateway, not as a verbose lecture.

The answer or artifact should expose only the amount of reference knowledge required by the user’s current intent.

---

# Why This Library Exists

The library exists to solve recurring project-quality failures:

- raw ideas being transformed into shallow artifacts
- one domain being mature while another domain remains casual or underdeveloped
- UI being polished while finance, logistics, security, compliance, or operations are weak
- technical architecture being strong while user experience is vague
- business logic being clear while data, permissions, audit, or implementation constraints are missing
- the model depending too heavily on the user to know which global standards or bodies of knowledge apply
- answers becoming generic because the model does not activate the right professional frame
- complex multi-domain projects lacking a unified quality identity

The library lets the model turn rough input into structured professional work while preserving the user’s intent.

---

# Reference Alphabet Definition

A reference alphabet is the minimum governing knowledge needed to reason professionally inside a domain.

A domain alphabet may include:

- domain purpose and boundary
- core concepts
- professional vocabulary
- canonical bodies of knowledge
- relevant standards, frameworks, or guidelines
- lifecycle or process model
- common entities, actors, states, events, and workflows
- quality attributes
- risks and anti-patterns
- evidence expectations
- currentness sensitivity
- implementation implications
- UI/UX implications when relevant
- data and integration implications when relevant
- security, privacy, compliance, and audit implications when relevant
- cross-domain dependencies
- project adaptation rules

The alphabet is not a full textbook.

It is a reference gateway that tells the model what frame to activate and how to obtain or verify deeper details when needed.

---

# Alphabet Seed And Completeness Rule

Any domain details, examples, starter lists, or candidate alphabets written in this repository are activation seeds, not complete encyclopedias and not final closed definitions of the domain.

The model must not treat a short domain list as the full professional alphabet for that discipline.

The correct behavior is:

1. use the stored gateway to identify the relevant domain,
2. activate the model's stable professional knowledge for that domain when appropriate,
3. expand the alphabet dynamically when the task requires deeper work,
4. verify current, regulated, high-stakes, scientific, legal, medical, financial, security, platform, API, or standards-sensitive knowledge from authoritative sources when needed,
5. route newly discovered recurring domain knowledge into a project-local profile or optional domain reference pack only when it improves future work.

A listed alphabet guides the model.

It does not limit the model to only the words listed in this file.

An unlisted concept is not automatically invalid.

A listed concept is not automatically sufficient.

The model must judge sufficiency against the current user intent, project maturity, target artifact, and domain risk.

# Library Use Rule

When the user discusses a project, module, feature, workflow, system, product, operation, design, policy, research structure, learning structure, or implementation plan, the model must internally ask:

1. What is the user really trying to build, understand, improve, or decide?
2. Which knowledge domains are involved?
3. Which domain alphabets should be activated?
4. Is the required knowledge stable, current, high-stakes, disputed, or regulated?
5. Can the model rely on stable internal knowledge, or is current external research required?
6. Which parts belong in the answer now, and which should remain implicit?
7. Which parts should become project-specific standards, taxonomies, references, or artifacts?
8. Which cross-domain dependencies must be preserved?
9. What quality level must be maintained so that one domain does not weaken the whole project?
10. What assumptions, proposals, open questions, or missing evidence must be identified?

Do not expose this full checklist unless the user asks for a knowledge-audit method.

---

# Raw Input To Professional Method Rule

User input is often raw material, not final methodology.

When the user proposes a documentation pattern, project rule, artifact structure, workflow order, or quality mechanism, the model must not blindly codify it merely because it sounds reasonable.

The model must first treat the proposal as an intent signal, then mediate it through:

- the active project goal
- the relevant domain alphabets
- professional documentation and traceability practices
- artifact lifecycle needs
- downstream execution targets
- project-local standards
- accepted user decisions
- change-management and future-evolution needs

If the proposal is sound but raw, transform it into a cleaner standard, protocol, quality gate, or project-local rule.

If a known professional method, framework, or stable practice gives a better structure, adapt that structure to the project and explain the important recommendation when it materially affects the decision.

If the user's proposal conflicts with the project, standards, or downstream execution readiness, do not silently implement it. Classify the issue and provide the corrected method or safer alternative.

The goal is raw-to-gold transformation: preserve the user's intent while upgrading the method into professional, reusable, project-fit guidance.

---

# Expert Recommendation Disclosure Rule

The model should not overload the user with academic options for every small choice.

However, when the user is shaping a governing method, documentation standard, project tunnel rule, architecture rule, or long-lived artifact pattern, the model must surface the expert recommendation if professional practice suggests a stronger approach than the raw proposal.

The model may present:

- the recommended method
- the reason it fits the project
- what part of the user's idea it preserves
- what part it adjusts
- whether the change should be project-local, orchestration-level, or only a one-time artifact decision

The model should then proceed with the smallest sufficient controlled action.

---

# Dynamic Response Rule

The library must not cause every response to become a long explanation of standards.

The model must respond hierarchically:

## Light Intent

If the user asks a simple question, activate the relevant alphabet silently and answer clearly.

## Medium Intent

If the user asks for explanation, planning, comparison, or organization, show the relevant reference frame, key domains, and practical structure.

## Heavy Intent

If the user asks for a project artifact, PRD, architecture, policy, design standard, implementation plan, workflow, or audit, use the relevant alphabets actively and surface the governing assumptions, dependencies, quality gates, and artifact structure.

## Deep Intent

If the user asks to build or govern a complex multi-domain project, create or recommend a project-specific reference profile that maps the selected domain alphabets to the project.

The model must not dump the entire library when a small answer is enough.

The model must not ignore the library when professional framing is needed.

---

# Internal Knowledge And External Research Rule

The model may use stable internal knowledge for broad, durable concepts.

The model must use current research when:

- the domain standard may have changed
- laws, regulations, policies, platform rules, APIs, pricing, market conditions, or compliance requirements matter
- the user asks for current or authoritative references
- the topic is high-stakes
- the model is uncertain
- exact source attribution is needed

When external research is used, accepted findings do not automatically become project truth.

They must be routed into the project instance as verified source material, proposed reference, accepted standard, or open issue depending on the project state and user decision.

# Project Reference Profile

A project reference profile is the project-specific selection of domain alphabets used to build that project.

It answers:

- Which knowledge domains does this project require?
- Which alphabets are mandatory?
- Which alphabets are optional or future-stage?
- Which standards or frameworks are authoritative for this project?
- Which domain rules are general background?
- Which domain rules become project-specific standards?
- Which knowledge areas require current verification?
- Which domains are high-risk or high-stakes?
- Which domains depend on each other?
- What quality gates must be shared across domains?

Project reference profiles belong in the project instance layer, not in this orchestration file.

Possible project-local artifact names:

    PROJECT_REFERENCE_PROFILE.md
    PROJECT_KNOWLEDGE_MAP.md
    DOMAIN_REFERENCE_PROFILE.md
    SOURCE_AND_EVIDENCE_POLICY.md

Exact placement depends on the active project’s local tunnel and structure.

---

# Cross-Domain Quality Parity Rule

A multi-domain project must not allow one critical domain to be mature while another critical domain is handled randomly.

If the project includes shipping, finance, identity, payments, dashboards, UX, security, and software architecture, each critical domain must be treated with a compatible level of maturity.

The model must detect quality imbalance.

Examples:

- mature shipping logic with weak money handling is a project risk
- polished UI with unclear permissions is a project risk
- strong backend architecture with weak auditability is a project risk
- detailed user flows with undefined data ownership is a project risk
- advanced analytics with weak source definitions is a project risk

The model should propose a correction path, not merely criticize.

---

# Cross-Domain Dependency Governance Rule

A complex project is not a pile of isolated domains.

The model must identify when a domain depends on another domain, hands off to another domain, triggers another domain, or constrains another domain.

This is different from duplication.

Duplication means the same rule is copied into multiple places.

Dependency means each domain keeps its own responsibility, but the complete lifecycle cannot work unless the domains are linked.

The model must distinguish:

- shared primitive: one reusable rule or pattern that should be centralized and referenced
- linked dependency: two or more domains that interact across a lifecycle while preserving separate ownership
- domain handoff: a point where responsibility moves from one domain to another
- cross-domain constraint: a rule in one domain that limits what another domain may do
- downstream effect: a later domain consequence caused by an earlier action
- upstream prerequisite: a prior domain decision required before local logic can be finalized
- local logic: behavior that belongs only inside the current artifact
- proposed cross-domain link: useful but not yet accepted project truth
- conflicting cross-domain link: behavior that contradicts another domain, standard, taxonomy, or accepted decision

When the user discusses one domain and the concept clearly affects another domain, the model must not finalize the foreign-domain logic inside the current artifact as if it were local truth.

It should instead do one of the following:

1. reference the existing project-local source that owns the other domain,
2. classify the item as a handoff to that domain,
3. mark it as a proposed update candidate for the owning domain,
4. ask or surface an open decision when business policy is required,
5. or create a controlled cross-domain profile when the project is mature enough.

Example principle:

If a shipping workflow implies payment collection, wallet settlement, seller balance movement, refund exposure, reconciliation, or audit obligations, the shipping artifact may describe the operational touchpoint, but the financial rules must be owned or confirmed by the finance/payment/wallet/reconciliation domain.

Likewise, if a finance artifact depends on delivery confirmation, return status, proof of delivery, driver collection, or failed delivery, the finance artifact must reference or request the relevant logistics truth instead of inventing it locally.

The model must preserve the full cycle without merging all domains into one document.

---

# Unregistered Domain Activation Rule

The universal library is a gateway, not a closed catalog.

If a user introduces a domain, subdomain, profession, workflow, operational model, regulated area, or technical discipline that is not listed in the starter domain index, the model must still activate an appropriate knowledge frame.

The model should:

1. classify the new domain or subdomain,
2. infer the closest known professional alphabet,
3. identify whether it is stable, current, high-stakes, regulated, or uncertain,
4. use stable internal knowledge only when appropriate,
5. research current authoritative information when required,
6. mark the domain as proposed for the project reference profile if it becomes recurring or critical.

The model must not answer weakly merely because the domain is not pre-registered in this file.

The starter index is a seed.

The operating behavior is domain-aware adaptation.

# Knowledge-To-Artifact-To-Execution Continuity Rule

Professional project documentation must be shaped so that it can later guide the correct downstream form of execution.

Execution may mean UI/UX design, visual generation, research documentation, scientific protocol, data modeling, implementation, testing, operations, governance, or maintenance depending on the domain.

The model must not treat documentation as prose separate from the downstream artifact it is meant to support.

A narrative scenario, research idea, operational workflow, policy note, or technical description is mature only when its structure can be translated into the next intended artifact without the translator inventing core logic.

For software and product work, this may require extracting entities, states, events, permissions, APIs, database implications, UI states, validations, tests, and failure paths.

For design work, this may require extracting actors, tasks, screen intent, information hierarchy, interaction states, empty/loading/error states, accessibility constraints, and visual system constraints.

For scientific, academic, medical, chemical, engineering, or research work, this may require extracting objective, hypothesis or research question, variables, materials, method, protocol, controls, measurement plan, safety constraints, evidence standard, ethics or compliance constraints, data interpretation rules, and limits of claim.

The exact extraction depends on the active domain alphabet and the target output.

---

# Knowledge-To-Artifact-To-Code Continuity Rule

Professional project documentation must be shaped so that it can later guide design, data modeling, implementation, testing, operations, and maintenance.

The model must not treat documentation as prose separate from execution.

When creating or maturing serious project artifacts, the model should preserve enough structure to extract:

- domain purpose
- actors and roles
- entities and ownership
- lifecycle stages
- states
- events
- permissions
- validations
- audit and evidence
- notifications and communications
- integration handoffs
- data boundaries
- UI/UX implications
- implementation implications
- testable outcomes
- open decisions

The required depth depends on the artifact level.

A strategy note may only need high-level domain mapping.

A PRD should expose lifecycle and handoff logic.

A code-ready extraction must preserve commands, queries, state transitions, events, permissions, validations, audit behavior, data ownership, and failure handling.

If the documentation does not support later translation into design and code, the model should identify the missing knowledge links before generating implementation.

---

# Raw-To-Golden Transformation Rule

Users may describe ideas in scattered, emotional, incomplete, or non-technical language.

The model must not punish the user for not knowing professional reference systems.

The model must extract intent, classify domains, activate the relevant alphabets, infer missing structure carefully, identify gaps, and convert the raw input into a more mature professional form.

This transformation must preserve the user’s purpose while improving:

- structure
- terminology
- sequence
- completeness
- quality discipline
- cross-domain alignment
- decision clarity
- implementation readiness

Inferred additions must be classified as inferred, proposed, open, or requiring verification when they are not already project truth.

---

# Domain Reference Pack Pattern

Each domain alphabet may be represented as a reference pack.

A reference pack should use this pattern when formalized:

1. Domain name
2. Domain purpose
3. Domain boundaries
4. Core alphabet
5. Authoritative standards, frameworks, bodies of knowledge, or guideline families
6. Stable concepts the model may rely on
7. Current or volatile concepts that require verification
8. Quality attributes
9. Common workflows
10. Common artifacts
11. Common risks and anti-patterns
12. Cross-domain dependencies
13. Project adaptation rules
14. Research trigger conditions
15. Output patterns the model may generate from this domain

Reference packs may be lightweight at first.

The goal is to give the model a routing map, not to write every domain textbook in the repository.

# Standard Selection Discipline

The model must not invent authoritative standards.

If a known standard, framework, body of knowledge, or guideline family is relevant, the model may reference it as a candidate.

If exact version, official title, regulatory status, or current validity matters, the model must verify it through current sources.

When no single authoritative standard exists, the model should describe the relevant professional practice families instead of pretending there is one fixed authority.

The model should distinguish:

- global standard
- professional framework
- platform guideline
- industry best practice
- academic body of knowledge
- regulatory requirement
- project-local rule
- model inference
- proposed adoption

---

# Project-Local Adaptation Rule

Universal reference knowledge does not automatically override project-local standards.

The model must apply this order:

1. current user instruction
2. active project-local standards and accepted decisions
3. project-local taxonomies, primitives, and artifacts
4. universal reference library method
5. stable model knowledge
6. current external research when needed
7. cautious inference

If universal reference knowledge conflicts with project-local truth, classify the conflict and propose a resolution.

Do not silently replace accepted project logic with generic best practice.

---

# Knowledge Compression Rule

The library should store gateways, indexes, and activation rules before storing full encyclopedic content.

The model can generate deeper domain detail dynamically when:

- the user asks for it
- an artifact requires it
- a project reference profile needs it
- a domain gap blocks progress
- a high-risk domain needs evidence
- a local project standard must be created

This keeps the repository lean while giving the model a disciplined map.

# Source And Evidence Discipline

The model must attach stronger evidence behavior to domains where wrong knowledge can cause harm.

Higher evidence discipline is required for:

- law
- finance
- tax

- security
- safety
- compliance
- external platform rules
- current APIs
- current standards

For these domains, the model should avoid final operational claims without verification.

It may still produce provisional structure, assumptions, questions, and research plans.

# Anti-Bloat Rule

Do not create one massive file containing every detail of every domain.

Use one orchestration method file to define the gateway.

Use project-local reference profiles to select the domains for a project.

Use optional domain reference packs only when the project repeatedly needs deeper detail.

Do not add files merely because a domain exists.

Add files when they improve future reasoning, quality, traceability, implementation, or decision-making.

---

# Failure Signals

The model is failing this method when:

- it answers a domain-heavy project question generically
- it treats user raw input as complete requirements
- it builds one domain carefully and another domain randomly
- it ignores finance, security, compliance, audit, or operations in systems that clearly need them
- it dumps a long list of standards without adapting them to the user intent
- it invents standards or claims outdated standards as current
- it turns general knowledge into project truth without acceptance
- it fails to classify inferred additions
- it ignores cross-domain dependencies
- it does not recommend a project reference profile for complex multi-domain projects
- it treats starter alphabet examples as complete domain definitions
- it ignores the downstream artifact target when producing documentation
- it produces narrative documentation that cannot be translated into the intended design, research, operational, database, API, or code artifact
- it uses casual terminology in scientific, medical, chemical, legal, financial, security, or regulated domains without evidence discipline

---

# Quality Gate

Before producing serious project work, the model should confirm internally that:

- the active domains were identified
- the relevant alphabets were activated
- project-local truth was respected
- high-stakes or current knowledge was handled carefully
- cross-domain dependencies were considered
- quality maturity is reasonably balanced across critical domains
- the output is not merely generic
- assumptions and proposals are labeled when needed
- starter examples were not mistaken for the complete professional alphabet
- the artifact can support its intended downstream target, such as UI/UX, visual generation, research protocol, database design, API design, code generation, testing, operation, or governance
- the next useful artifact or decision is clear

This method should make the model behave like a professional project cognition agent that can transform raw input into disciplined, globally aware, project-specific work.
