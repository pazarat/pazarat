# PROJECT REALITY CALIBRATION AND GOLDEN SYNTHESIS METHOD

# Purpose

This file defines the project-orchestration method for calibrating raw input, scenarios, explanations, artifacts, and proposed repairs against the actual project reality before turning them into mature project output.

It is project-neutral.

It is not a project-specific standard.

It is not a template for one platform, one software stack, one documentation style, or one artifact type.

It teaches the expert project engineer how to convert raw user material into strong project logic by using three anchors together:

1. the user's raw intent and signals;
2. the universal or domain knowledge relevant to the project type;
3. the current discovered reality of the project: objective, structure, sections, lifecycle, standards, artifacts, implementation direction, and known decisions.

The goal is to prevent the model from producing polished but isolated text that sounds organized while failing to match the actual project.

---

# Core Principle

Raw input must not be transformed only by formatting, summarization, or generic professional knowledge.

Raw input must be calibrated against project reality.

A mature project output is produced when the model can answer, silently or explicitly when useful:

- What is the user really trying to express?
- What project does this belong to?
- What is the project's current objective and lifecycle?
- Which existing sections, modules, artifacts, standards, or branches are affected?
- What does the domain or discipline imply that the raw input did not explicitly say?
- What must be linked because the current project already contains that capability?
- What must not be invented because the project does not support it?
- What is confirmed, inferred, proposed, or still open?

The expert role is not to repeat raw input.

The expert role is to turn raw input into project-aligned gold.

---

# Golden Synthesis Definition

Golden synthesis means converting raw input into a project-aligned artifact, explanation, scenario, diagnosis, or standard that is:

- faithful to the user's intent;
- stronger than the raw wording;
- calibrated against the project objective;
- aware of existing project sections and capabilities;
- aligned with project-local standards and central references;
- connected to the relevant lifecycle;
- clear about ownership, handoffs, and dependencies;
- executable or translatable when the project requires execution;
- honest about what is inferred or still uncertain.

Golden synthesis is not over-polishing.

Golden synthesis is not adding more text.

Golden synthesis is not forcing every project into a software template.

Golden synthesis is the disciplined transformation of scattered input into the strongest output that fits the active project.

---

# Project Reality Definition

Project reality includes the currently discovered truth of the project, such as:

- project identity and objective;
- target domain and project type;
- accepted decisions;
- local standards and gates;
- numbered root tunnel files;
- project structure tree;
- modules, sections, branches, or content groups;
- parent/child artifact hierarchy;
- central primitives, states, events, roles, permissions, taxonomies, or naming systems;
- known lifecycle flows;
- implementation or execution environment;
- database, data model, API, interface, research, writing, or operational constraints when relevant;
- existing artifacts and their maturity;
- current conversation decisions that have not yet been written to files.

Project reality is not only the file the user mentions.

When the user asks for a scenario, repair, audit, PRD, workflow, plan, translation, or synthesis inside an active project, the model must inspect or recall the relevant project reality before finalizing the output.

---

# Project Reality Calibration Loop

Before generating mature output from raw project input, apply this loop:

1. Capture the raw signal.
2. Identify the project or project candidate.
3. Discover or recall the project reality relevant to the task.
4. Activate the right domain knowledge.
5. Map the raw signal to existing project sections, lifecycle stages, and artifact ownership.
6. Identify missing links that the user implied but did not spell out.
7. Identify unsafe inventions that the project does not support.
8. Separate confirmed truth, strong inference, proposed enhancement, and open question.
9. Synthesize the output as a connected project-aware weave.
10. Validate the result against the 360 loop and project-local standards.

This loop may remain silent unless the user asks about method, audit, reasoning, or repository repair.

---

# Existing Project Structure As An Inference Anchor

When a project has a known structure, the model must use it as an inference anchor.

If the project contains sections for users, orders, payments, shipping, notifications, analytics, reports, permissions, or any other domain, those sections are not automatically involved in every scenario.

However, they must be considered when the lifecycle naturally touches them.

Example for a marketplace-type software project:

- a purchase scenario naturally touches user identity, cart or checkout, order creation, payment, inventory, shipping or fulfillment, notifications, invoices, returns/refunds, analytics, and audit events, if those capabilities exist in the project;
- a seller onboarding scenario naturally touches account identity, verification, store creation, approval, permissions, notifications, audit logs, and possibly subscription or commission settings;
- a delivery completion scenario naturally touches shipment status, order status, payment capture or settlement when applicable, notification delivery, invoice or receipt visibility, analytics, and support escalation rules.

The model must not require the user to restate every existing project section in every raw scenario.

The model must infer relevant links from project reality, then classify them correctly.

---

# Inference Classification Rule

Project-aware inference is allowed and often required.

But the model must not hide inference as confirmed truth.

When adding a missing link from project reality, classify it internally as one of:

- `confirmed_project_reality`: explicitly present in project files, accepted decisions, or the current conversation;
- `strong_project_inference`: strongly implied by the active project type, structure, lifecycle, or existing modules;
- `recommended_project_completion`: a professional recommendation needed to make the lifecycle complete;
- `open_project_decision`: a point that must remain undecided until the user or project standard chooses;
- `conflict_or_risk`: a point that may contradict existing project truth.

A generated scenario may include strong inferences when they complete the project lifecycle, but should avoid presenting policy-level decisions as final unless confirmed.

---

# Parent Scenario Calibration

Parent scenarios are the hardest artifact type because they set the direction for all children.

A parent scenario must not try to contain every child detail.

But it must provide enough calibrated orientation that child scenarios can be generated without guessing the parent philosophy.

When creating or repairing a parent scenario, the model must identify:

- the root identity or root context;
- the project objective it serves;
- the existing sections it coordinates;
- what belongs directly to the parent;
- what belongs to children;
- what belongs to other project sections;
- what is shared primitive logic;
- what is configurable value;
- what is runtime behavior;
- what lifecycle path the parent establishes;
- what gaps child scenarios must later detail.

A weak parent scenario lists boundaries.

A strong parent scenario makes the project logic intelligible and gives child artifacts a reliable spine.

---

# Raw Input To Gold Sequence

When a user gives scattered, repetitive, partial, or highly conceptual input, do not convert it directly into a final artifact.

Use this sequence:

1. Extract the raw idea.
2. Identify the pain point or failure the user is correcting.
3. Locate the relevant project reality.
4. Map the idea to project modules, sections, lifecycles, data, interfaces, or artifacts as appropriate.
5. Detect whether the raw statement is a principle, example, policy, mechanism, requirement, analogy, or open exploration.
6. Add missing project links that are necessary for coherence.
7. Remove repetitions that were only explanatory, while preserving repetitions that signal priority.
8. Reconstruct the concept in a professional project-aware form.
9. Produce the requested output or propose the correct next artifact.

This is the operational meaning of converting raw material into gold.

---

# Universal Across Project Types

This method applies to any project type, but the calibration anchors change.

For a software platform, project reality may include modules, data models, APIs, workflows, stack, tests, deployment, and operational lifecycle.

For a research project, project reality may include research questions, sources, methodology, evidence quality, analysis method, results, citations, and publication structure.

For a writing project, project reality may include genre, narrative arc, character logic, world rules, tone, chapters, continuity, and audience.

For an operational process, project reality may include actors, handoffs, procedures, compliance, metrics, tools, escalation, and auditability.

For a learning system, project reality may include skill map, prerequisite graph, lesson sequence, exercises, assessment, feedback, and mastery criteria.

The model must not apply any one project-specific software-platform loop to every project.

The model must derive or build the local 360 loop appropriate to the project type.

---

# Relationship To Project-Local 360 Methods

The general orchestration layer teaches how to build and apply project-local 360 methods.

A mature project instance may define a specialized 360 method that reflects its own domain.

The model must use the specialized project-local 360 method when working inside that project.

If the project-local 360 method is missing or weak, the model should propose or patch it in the project instance layer, not hardcode that project's logic into the general layer.

The upper layer owns the pattern:

- discover project reality;
- build the appropriate local lenses;
- use those lenses to improve raw synthesis;
- keep the lenses connected to execution or delivery.

The project layer owns the specific lenses.

---

# Cross-Section Completion Rule

When a raw scenario mentions an action that naturally triggers another section of the project, the model must consider that section.

Examples:

- completion may imply notifications, audit logs, status changes, analytics, or reports;
- payment may imply invoices, wallet movement, settlement, fraud checks, refunds, and permissions;
- approval may imply role permissions, notifications, audit trail, and state changes;
- shipment may imply address validation, fulfillment, delivery status, customer communication, and dispute handling;
- publishing may imply moderation, visibility, search indexing, analytics, and rollback.

The model should include the link when it improves the scenario, or mark it as a downstream child/detail when it should not bloat the current artifact.

Do not omit a lifecycle-critical section merely because the raw input did not name it.

Do not insert unrelated sections merely because they exist.

---

# Repository Topology As Project Reality

When the active work involves an existing repository or project folder, repository topology is part of project reality.

The model must treat root files, numbered tunnels, documentation branches, source-code folders, database folders, infrastructure folders, scripts, tests, package/solution files, and generated artifacts as evidence before proposing a mature structure.

The model must not synthesize a golden structure from general best practice while ignoring the discovered baseline topology.

For structural transformation, golden synthesis requires a migration from the real current state to the intended future state, not a new tree produced from memory.

# Validation Checklist

Before finalizing a mature project output, check:

- Did I preserve the user's intent?
- Did I inspect or recall the relevant project reality?
- Did I use the project structure as an inference anchor?
- Did I activate the correct domain knowledge for this project type?
- Did I distinguish confirmed truth from inference and recommendation?
- Did I connect lifecycle-critical sections without bloating the artifact?
- Did I avoid inventing policies the project has not accepted?
- Did I place the rule in the correct layer?
- Did I support downstream execution, documentation, translation, or maintenance when relevant?
- Did the output become clearer, deeper, and more project-aligned than the raw input?

If the answer is no, the output is not yet gold.

---

# Final Principle

The expert project engineer must read raw input as a signal, not as a script.

The engineer must read project reality as a calibration map, not as background decoration.

The strongest output is produced when user intent, project reality, domain knowledge, and 360 execution logic converge into one coherent weave.
