# Assumption and Contradiction Guard — مانع الافتراضات والتناقضات

Pazarat work must be grounded in documented reality, not in assumptions or guesses about file structure, content organization, or rule application.

This guard prevents the model from treating inferred structure as documented fact, allowing breaches of stated rules without triggering escalation, or generating output based on guesses rather than verification.

## What This Guard Prevents

1. **Guessing at Structure**
   - Don't: "The User_Management_PRD contains Section X for State and Event Alignment"
   - Don't assume section names exist without reading the file
   - Do: Read the file and verify; mark as "contains," "references," "organized as," or "missing"

2. **Treating Assumptions as Rules**
   - Don't: "The system enforces X" when the enforcement mechanism wasn't verified to be active
   - Don't claim the repository prevents guessing if guessing just occurred
   - Do: State enforcement only after verifying the guard actually fired and stopped the violation

3. **Analyzing Without Acting**
   - Don't: Respond to "solve this problem" with analysis only
   - Don't identify a root cause without proposing or applying a fix
   - Do: Analyze, identify, propose patch, apply patch, test, continue

4. **Allowing Rule Violations Without Escalation**
   - Don't: Treat a breach of "evidence discipline" as a minor output issue
   - Don't continue as if nothing happened when state/event naming rules are violated
   - Do: Flag the breach, classify it, propose a layer fix, and apply it

5. **Skipping Verification of Core Claims**
   - Don't: Say "the parent PRD defines X" without checking the actual parent content
   - Don't reference documented rules as governing a decision when not verified
   - Do: Before major claims, verify the source actually contains what is claimed

## Active Guards for Pazarat Work

Before generating content about Pazarat artifacts:

**Document Existence Check:**
- Claim: "User_Management_PRD.md has section X"
- Guard: Read the file and verify the section exists
- Action: If verified → state "contains"; If not found → state "missing" or "organized as Y"

**Rule Enforcement Verification:**
- Claim: "The repository enforces X discipline"
- Guard: Verify the enforcement mechanism exists and is actually being applied
- Action: If enforcement rule exists but wasn't applied → escalate as "enforcement gap"

**Parent-Child Consistency Check:**
- Claim: "Child should inherit X from parent"
- Guard: Read the actual parent to confirm X is documented there
- Action: If parent doesn't contain X → mark as "parent candidate" not "parent inheritance"

**State/Event Terminology Check:**
- Claim: "Use state family X from 02_PLATFORM_STATE_TAXONOMY.md"
- Guard: Read the taxonomy and verify state family X exists
- Action: If not found → mark as "proposed state" not "documented state"

**Contradiction Detection:**
- Contradiction: "Evidence discipline forbids guessing, but I just guessed"
- Guard: Classify the contradiction and identify where enforcement failed
- Action: Propose a patch to the failing guard, apply it, update project memory

## Guard Activation Checklist

Use this before committing to major output:

- [ ] Am I referencing a documented artifact? If yes, have I read its current content?
- [ ] Am I claiming a rule is enforced? If yes, did I verify the enforcement mechanism is active?
- [ ] Am I proposing a change? If yes, did I check the actual current state?
- [ ] Am I generating derivative content (child, workflow, etc.)? If yes, did I read the source?
- [ ] Did I detect that a rule should have prevented this action? If yes, have I escalated it?
- [ ] Is the user asking for a decision? If yes, am I proposing a concrete fix, not just analysis?
- [ ] Have I classified my claims as "documented," "proposed," "inferred," or "missing"?

If any check fails, pause and verify before continuing.
