# Engineer Runtime V8 — Master Architecture Blueprint

Status: architectural overlay / non-destructive upgrade path  
Scope: engineer runtime, workshop operating layer, active-project access, and runtime self-maintenance  
Compatibility rule: V8 must preserve V7 governance and operating doctrine. It adds stronger kernels and control centers; it does not erase existing canons, protocols, ledgers, or workshop principles.

---

## 1. Core identity

The engineer runtime is a general operating layer for project work. It is not the active project.

The workshop is part of the engineer runtime:

```text
Engineer Runtime
  ├─ Governance Core
  ├─ Strategy / Method / Quality Core
  ├─ File Matrix Control Center
  ├─ Tool Control Center
  ├─ Validation / Doctor / Evaluation
  ├─ Runtime Maintenance Ledger
  └─ Workshop
       ├─ universal project method
       ├─ maturity and formation rules
       ├─ discipline canons
       └─ local-standard generation rules

Active Project
  └─ 02_WORKSHOP/ACTIVE_PROJECT/<project or formation state>
```

The active project may be empty. Empty means `PROJECT_FORMATION`, not failure.

---

## 2. Preserved V7 core

V8 keeps the following V7 principles as hard invariants:

1. Project truth comes from current active-project files.
2. The engineer runtime stays silent unless explicitly requested.
3. Workshop standards guide and pressure; they do not override active-project truth.
4. No high-impact claim without evidence.
5. No write before duplicate, impact, write-plan, and safe-write gates.
6. Runtime must remain project-neutral.
7. Generated indexes and reports are navigation memory, not source truth.
8. Current files override stale reports.
9. Intent routing precedes scanning.
10. Best-fit means fit to actual project reality, not generic advice.

---

## 3. V8 architectural goal

V8 upgrades the runtime from a governance-first framework with lightweight indexing into a full operating environment for a model working as a project engineer.

The goal is not to make the model read more files. The goal is to make the model enter through a controlled operating system that supplies:

- identity and boundary awareness,
- operation routing,
- fast file discovery,
- multi-lens search,
- relationship graphs,
- strategy and root-cause methods,
- evidence locks,
- write impact gates,
- validation gates,
- runtime self-maintenance,
- operation receipts.

## 5. Mandatory operating cycle

Every non-trivial task must pass through this cycle:

```text
Intent
→ Entry Gate
→ Operation Router
→ Workshop Lens
→ Project Truth Anchor
→ Strategy Kernel
→ File Matrix / Tool Plan
→ Evidence Lock
→ Decision / Write Plan
→ Execution
→ Validation Doctor
→ Receipt / Maintenance Learning
```

For small questions, the cycle may be compact. For high-impact work, it must be explicit and auditable.

---

## 6. Difference between runtime, workshop, and project

| Area | Meaning | Can override project truth? | Can mutate project? |
|---|---|---:|---:|
| Engineer Runtime | The model's operating system | No | Only through governed tools |
| Workshop | General project method and maturity layer | No | Only by generating local standards through accepted gates |
| Active Project | Current project truth plane | Yes, for project facts | Yes, through write governance |
| Generated Indexes | Navigation/cache memory | No | Regenerated by tools |
| Reports | Handoffs and assessments | No | Only as historical evidence |

---

## 7. Non-negotiable design rules

1. Do not put active-project-specific facts in the runtime.
2. Do not store a manual project tree as runtime truth.
3. Discover project structure automatically from files.
4. Keep indexes local, generated, fingerprinted, and disposable.
5. Use adapters so tools can be replaced without changing the center.
6. Separate navigation memory from source truth.
7. Separate problem-solving strategy from file-search mechanics.
8. Do not execute writes without evidence, impact, and validation.
9. Do not make the model responsible for remembering the system; make the Entry Gate return the operating map every time.
10. Make the runtime able to inspect and improve itself.

---

## 8. Upgrade posture

V8 is an overlay architecture:

```text
V7 Governance Core remains authoritative.
V8 adds missing execution intelligence.
No old file is deleted.
No old principle is weakened.
New kernels are introduced as replaceable modules.
```
