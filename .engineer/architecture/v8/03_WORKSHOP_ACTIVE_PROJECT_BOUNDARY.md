# V8 Workshop and Active Project Boundary

Status: mandatory boundary contract  
Purpose: keep the engineer runtime, workshop method, and active project truth separated while allowing them to cooperate.

---

## 1. Runtime topology

```text
Repository Root
│
├─ .engineer/
│   └─ engineer runtime, governance, tools, indexes, ledgers, architecture
│
├─ 02_WORKSHOP/
│   ├─ _WORKSHOP_SYSTEM/
│   │   └─ universal project method and discipline canons
│   │
│   └─ ACTIVE_PROJECT/
│       └─ current project truth or formation state
```

---

## 2. Boundary rules

1. `.engineer/` is the operating system for the model.
2. `02_WORKSHOP/_WORKSHOP_SYSTEM/` is part of the engineer operating environment.
3. `02_WORKSHOP/ACTIVE_PROJECT/` is the only active-project truth plane.
4. The workshop may evaluate, mature, and generate local-standard needs.
5. The workshop cannot override active project truth.
6. The runtime cannot hardcode active project facts.
7. The active project may be empty; empty means formation mode.
8. Generated runtime state cannot become project truth.

---

## 3. Operating sequence

```text
enter engineer runtime
→ identify operation target: runtime / workshop / active project
→ apply workshop method silently when project work is requested
→ anchor to active project truth
→ run file matrix / strategy / evidence gates
→ execute only through accepted project-local truth
```

---

## 4. Active-project states

```text
PROJECT_FORMATION
  Active project has no meaningful truth yet.
  Action: capture baseline and create accepted source truth.

PROJECT_DISCOVERY
  Project has truth files but incomplete execution surface.
  Action: map, mature, and avoid implementation from thin truth.

PROJECT_DOCUMENTED
  Requirements, standards, or artifacts exist.
  Action: use owner files and cross-artifact checks.

PROJECT_IMPLEMENTATION
  Code, tests, config, APIs, or database files exist.
  Action: activate File Matrix code intelligence and validation.

PROJECT_MAINTENANCE
  Existing implementation requires repair or evolution.
  Action: impact, regression, validation, and decision records.
```

---

## 5. Workshop responsibilities

The workshop supplies:

- universal 360-degree method,
- maturity assessment canons,
- formation baseline rules,
- discipline-specific canons,
- project-neutral evaluation pressure,
- local-standard generation rules,
- incomplete-truth detection.

The workshop does not supply project facts.

---

## 6. Local standard generation

When the workshop detects a missing project-local standard, it cannot execute directly. It must request or generate a local standard through governance:

```text
workshop detects gap
→ Strategy Kernel frames gap
→ File Matrix finds owner candidates
→ Write Impact Gate checks duplication and impact
→ local standard is proposed or written into active project truth
→ project truth becomes the execution authority
```

---

## 7. Maintenance ledger

The runtime maintenance ledger records engineer-runtime evolution only.

It must not override:

- active project files,
- current runtime contracts,
- current test results,
- current doctor results.

It may preserve architectural decisions and warnings for future sessions.
