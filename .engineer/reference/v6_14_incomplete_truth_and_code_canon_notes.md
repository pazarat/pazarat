# v6.14 Incomplete Truth And Code Canon Notes

v6.14 adds a truth-maturity layer to the engineer runtime.

## Why

Previous versions could find evidence, compare gaps, and route operations. The missing distinction was:

- a missing file is a gap;
- an existing but weak authoritative file is **incomplete truth**.

The engineer must not bypass incomplete truth, but must not blindly implement over it either.

## Added behavior

- `truth-maturity` classifies owner evidence as accepted, partial, incomplete, missing, or conflict-prone.
- `canon-check` detects whether code generation has a practice canon beyond stack selection.
- `plug` includes `truth_maturity_summary` and `canon_check_summary` for relevant operations.
- the neural graph includes `incomplete_truth_reflex` and `code_practice_reflex`.
- The active project snapshot receives `18_PAZARAT_CODE_PRACTICE_AND_IMPLEMENTATION_CANON.md` and new runtime gates.

## Core rule

Discovery may identify and propose repair for incomplete truth. Production-oriented execution must wait for accepted project truth or an explicit decision.
