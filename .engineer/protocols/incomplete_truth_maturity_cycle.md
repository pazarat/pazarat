# Incomplete Truth Maturity Cycle

This protocol converts the idea of **incomplete truth** into an operating cycle for the engineer runtime.

## Purpose

The engineer must mature project truth without bypassing it. A reference file can be authoritative but still too weak for a requested execution. The right action is neither blind obedience nor invention; it is classification, owner repair, and gated execution.

## Cycle

```txt
Request
→ plug / cycle
→ route
→ locate owner truth
→ compare to higher truth and operation requirements
→ classify: accepted / incomplete / missing / conflict / proposed repair
→ if incomplete: repair owner or decision record
→ only then execute production-oriented work
```

## Non-negotiable distinction

Discovery may say:

```txt
This truth is incomplete and needs repair.
```

Execution may not say:

```txt
I will implement the repair as if it were already accepted truth.
```

## Required checks

- Is there an owner file?
- Is it central, parent, child, local, or generated?
- Does higher project truth demand more than the owner currently states?
- Does the next layer require extractable entities, states, events, permissions, API, data, tests, or code practice rules?
- Would proceeding create hidden invention?
- Is the repair an update, a new central standard, or an explicit decision?

## Output shape

Every incomplete truth finding should be expressed as:

```txt
Owner: ...
Status: incomplete_truth
Missing dimensions: ...
Evidence: ...
Impact: ...
Repair recommendation: ...
Execution allowed: no / only for discovery / yes after accepted repair
```
