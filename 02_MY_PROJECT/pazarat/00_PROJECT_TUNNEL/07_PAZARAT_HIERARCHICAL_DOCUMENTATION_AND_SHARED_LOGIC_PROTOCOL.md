# PAZARAT HIERARCHICAL DOCUMENTATION AND SHARED LOGIC PROTOCOL

# Purpose

This file defines the Pazarat-specific protocol for hierarchical documentation, parent-child scenario generation, source scenario preservation, coverage mapping, shared logic handling, and controlled PRD evolution.

It belongs to the Pazarat project-local numbered tunnel.

It is not a general project orchestration file.

It is not a cognitive runtime file.

It is not a replacement for Pazarat identity, taxonomy, event taxonomy, structure index, UI standard, or implementation standard.

Its purpose is to make Pazarat PRD work cumulative, traceable, non-generic, and safe to expand from parent scenarios into child scenarios without losing accepted project meaning.

---

# Core Principle

Pazarat documentation must grow hierarchically.

A parent artifact defines the broad area.

A child artifact expands one focused branch.

Shared logic must not be repeated inconsistently.

Raw source scenarios must not be erased before useful signal is extracted.

Every generated or revised Pazarat PRD should preserve project-local truth, parent context, child intent, state/event consistency, and implementation awareness.

The model must not generate polished but disconnected PRDs.

---

# Documentation Contract Binding Rule

Hierarchical documentation must be written through the Pazarat PRD documentation contract.

The governing local contract is:

```txt
11_PAZARAT_PRD_DOCUMENTATION_AND_IMPLEMENTATION_TRANSLATION_CONTRACT.md
```

Parent PRDs must remain section-level routing ledgers.

Child PRDs must deepen local executable logic.

Both must preserve traceability, translation readiness, and cumulative maintenance safety.

The model must not treat the contract as a cosmetic template.

It is the project-local mechanism that keeps narrative, hierarchy, routing, UI/UX readiness, implementation readiness, and later change control aligned.

# Parent PRD Role

A parent PRD defines a broad Pazarat area.

A parent PRD should own:

- area purpose
- area scope
- business or operational role
- major user/admin/system responsibilities
- high-level flows
- main child routes
- shared assumptions
- relevant states
- relevant events
- data and visibility boundaries
- permissions at a high level
- parent-level risks
- child roadmap
- open questions

A parent PRD should not contain every child detail.

It should create the conceptual and operational frame for children.

# Child PRD Role

A child PRD expands one focused route, module, screen family, workflow, or artifact under a parent.

A child PRD should own:

- local purpose
- inherited parent context
- local scope
- local non-scope
- target users or admin roles
- local workflow
- local states and events
- local permissions
- local validations
- local table/list behavior if relevant
- local actions
- local exceptions
- related shared primitives
- implementation implications
- open questions
- acceptance considerations

A child PRD must not redefine the whole parent area.

It must inherit from parent and specialize locally.

# Documentation As Collaborative Artifact
A Pazarat documentation artifact is not a template to fill; it is a collaborative knowledge map.
When generating or reviewing a PRD, the model must treat the process as:

- asking the user clarifying questions about scope, constraints, and goals
- proposing structure and content based on parent context and platform standards
- suggesting how the artifact connects to other platform areas
- offering domain-informed recommendations for workflow, states, events, and UI patterns
- building the document incrementally with the user
- exposing gaps that need child artifacts, taxonomy updates, or design decisions

The model should guide the user toward completeness while remaining fully accountable to project standards and the user's vision.
This collaborative approach strengthens hierarchy and ensures consistency.

# Child Build Order And Cross-Artifact Routing Rule

Before expanding a parent PRD into child PRDs, the model must not rely on folder order alone.

It must apply:

    10_PAZARAT_PARENT_CHILD_SEQUENCE_AND_CROSS_ARTIFACT_ROUTING_STANDARD.md

The model should identify:

- local child candidates
- prerequisite child candidates
- external artifacts mentioned by the parent
- sibling dependencies
- parent-to-parent relationships
- shared primitive candidates
- taxonomy candidates
- open routing decisions

The model should then choose the build order that best preserves narrative sequence, prerequisite logic, dependency ownership, design readiness, and implementation readiness.

A child-like concept mentioned inside a parent does not automatically belong in the current folder.

If the concept belongs outside the current branch, preserve it as an external route or seed reference instead of creating a misplaced child.

---

# Parent Routing Ledger Requirement

A mature parent PRD must act as the section heart, first-view map, and routing ledger.

It should preserve enough parent-level knowledge for the model, user, designer, developer, or future AI to understand:

- what children exist or are expected
- which child should be built first
- why that sequence exists
- which child depends on another child
- which concepts are external routes rather than local children
- which other parent PRDs are related
- which shared primitives, states, events, permissions, UI outputs, or implementation outputs may be affected

The parent should distribute responsibility, not consume all responsibility.

It must not write the full child logic, but it must preserve the routing reason that lets the child be written later without guessing.

When a parent mentions a login page, verification template, product display, payout flow, wallet behavior, permission model, or any concept owned elsewhere, the parent must record the relationship and expected owner instead of silently ignoring it or creating a misplaced child.

---

# Parent Routing Is Continuous Rule

Parent routing is not performed once at parent creation and then forgotten.

Every meaningful parent revision, child generation, sibling audit, cross-system discovery, or translation-readiness review should check whether the parent routing ledger remains accurate.

If new child ordering, child dependency, external ownership, parent-to-parent relationship, shared primitive, taxonomy candidate, or translation bridge is discovered, the model must either:

- update the parent through a controlled patch when the parent is in scope
- record a parent update candidate when the parent is not in scope
- classify the discovery as child-local when it should not affect the parent

This keeps the parent alive as the section map and prevents children from becoming disconnected isolated files.

# No Filler Rule

Pazarat PRDs must not be filled with generic product language.

Avoid:

- generic marketplace statements that do not affect the target artifact
- repeated broad platform claims in every child
- vague statements such as "improves user experience" without local meaning
- decorative lists that do not guide implementation or documentation
- invented features not grounded in project context
- repeated definitions that belong in shared primitives
- final-sounding requirements based on weak input

Every section should either clarify scope, guide behavior, preserve a decision, define a workflow, identify data/state/event logic, or support future implementation.

---

# Useful Repetition Versus Harmful Duplication

Some repetition is useful reinforcement.

Some repetition creates contradiction.

Useful repetition:

- reminds child artifacts of inherited parent context
- repeats critical safety or permission constraints
- preserves key Pazarat principles
- restates local scope boundaries

Harmful duplication:

- redefines shared concepts differently
- copies parent text into every child without specialization
- repeats generic platform identity in every artifact
- creates multiple versions of the same rule
- embeds shared primitives in local artifacts instead of referencing them

The model must preserve useful reinforcement and remove harmful duplication.

# State Taxonomy Alignment

Pazarat child PRDs must align with the platform state taxonomy.

When a child artifact introduces a status or state, the model must check whether it:

- already exists in state taxonomy
- is a local state under a broader platform state
- is a proposed new state
- conflicts with an existing state
- needs definition
- needs transition rules
- affects implementation or UI

Do not invent new states casually.

If a new state is necessary, mark it as proposed and identify affected artifacts.

## Mandatory Child Scenario Dimensions

A mature Pazarat child scenario should account for the following dimensions.

The exact headings may adapt to the artifact, but the substance must be present when relevant.

### 1. Artifact Identity

Identify:

- artifact name
- artifact type
- Pazarat branch
- parent PRD
- child route
- target folder or file
- related standards
- related sibling artifacts when relevant

This prevents orphan artifacts and wrong-path generation.

### 2. Parent Inheritance

Declare what the child inherits from the parent:

- parent purpose
- parent scope
- parent boundaries
- inherited entities
- inherited state families
- inherited event families
- inherited permissions
- inherited UI implications
- inherited implementation implications
- inherited shared primitives
- inherited open questions when relevant

The child must specialize inherited context.

It must not restart or redefine the parent.

### 3. Local Purpose

Explain why this child exists.

The local purpose should answer:

- what local problem this child solves
- what operational need it supports
- why it deserves a separate child artifact
- what would be unclear if this child did not exist

Avoid broad platform language that does not affect the local child.

### 4. Local Scope And Boundaries

Clarify:

- what this child owns
- what it references but does not own
- what remains parent-owned
- what belongs to siblings
- what belongs to another module
- what belongs to UI/screen artifacts
- what belongs to implementation notes
- what remains open

This prevents the child from swallowing parent, sibling, or external-domain responsibilities.

### 5. Source Scenario Preservation

If the child comes from raw user input, notes, or a placeholder, preserve the useful signal.

Extract:

- original intent
- key terms
- user/admin/system actions
- implied workflow
- implied states
- implied events
- implied permissions
- implied UI needs
- implied implementation needs
- ambiguous or unresolved points

Do not erase the original product meaning through over-polishing.

### 6. Local Scenario Narrative

Write the local operational story as connected product logic.

The narrative should explain:

- entry point
- actor
- context
- action
- system behavior
- decision point
- result
- state change
- event implication
- UI implication
- implementation implication
- boundary or handoff

The narrative should be readable by product owner, designer, developer, and future AI.

### 7. Local Workflow

Define the local workflow when the scenario has sequence.

Include when relevant:

- trigger
- actors
- preconditions
- steps
- decisions
- outcomes
- state transitions
- emitted events
- validations
- exceptions
- failure paths
- handoffs
- completion conditions

A workflow-heavy child may later need a separate workflow artifact.

### 8. State Alignment

Any state mentioned or implied must align with Pazarat state taxonomy.

Classify each state as:

- existing platform state
- local child state
- proposed new state
- conflicting state
- state requiring taxonomy update

The model must not invent states casually.

State logic must not remain hidden only inside prose.

### 9. Event Alignment

Any event, trigger, audit event, notification trigger, analytics event, or automation trigger must align with Pazarat event taxonomy.

Classify each event as:

- existing platform event
- local child event
- proposed new event
- conflicting event
- event requiring taxonomy update

The model must not invent event names casually.

Event logic should identify trigger, actor, target, effect, and downstream implications when relevant.

### 10. Shared Primitive Usage

Identify relevant shared primitives or primitive candidates.

Examples:

- user lifecycle
- verification
- approval
- rejection
- suspension
- profile ownership
- permission gate
- visibility
- audit log
- notification
- dashboard table behavior
- bulk action behavior
- document review
- event emission

Do not redefine shared logic differently inside the child.

If repeated logic appears but is not documented yet, classify it as a primitive candidate.

### 11. Permissions And Visibility

Clarify who can do what.

When relevant, identify:

- who can view
- who can create
- who can edit
- who can approve
- who can reject
- who can suspend
- who can reactivate
- who can assign
- who can export
- who can delete
- who can see sensitive fields
- which actions are hidden
- which actions are disabled
- which actions require confirmation
- which actions require audit

Avoid vague phrases such as "admin can manage" when the action set matters.

### 12. Audit And Accountability

When the child affects important records, identities, permissions, statuses, money, documents, orders, shipments, returns, settings, or security, define audit implications.

Include when relevant:

- actor
- action
- target entity
- before state
- after state
- reason/comment
- timestamp need
- source module
- related event

# Anti-Drift Rule

The model must not allow Pazarat documentation to drift away from accepted standards.

Drift may occur when:

- child files redefine shared logic differently
- old paths remain after migration
- parent and child disagree

- generic marketplace templates replace Pazarat logic
- raw source is erased before extraction
- implementation assumptions appear without traceability
- UI details conflict with UI standard
- generated artifacts ignore prior conversation decisions

When drift is detected, classify it and recommend correction.
