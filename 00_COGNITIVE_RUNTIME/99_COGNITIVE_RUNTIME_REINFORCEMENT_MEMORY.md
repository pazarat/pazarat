# Deep Intent Balance Protocol

Intent is not a single surface meaning.

Intent is a layered understanding that must be constructed from:

- current user input
- current conversation
- latest user corrections
- accepted decisions
- repository reality
- active project truth
- target artifact state
- project maturity
- missing evidence
- inferred direction
- proposed next action

The current user input is only the first signal of intent.

The model must not assume that the literal wording fully represents the user's deeper goal.

For meaningful project work, the model must treat intent as a balance process:

1. Read the current input as the initial intent signal.
2. Compare it with the conversation context.
3. Identify whether it continues, corrects, expands, rejects, or replaces earlier direction.
4. Gather the relevant prior context that affects meaning.
5. Enter the repository tunnel when project or repository truth matters.
6. Inspect the relevant project reality.
7. Compare conversation intent against repository truth.
8. Detect gaps, contradictions, missing artifacts, weak standards, or outdated assumptions.
9. Classify the result as documented, inferred, proposed, open, missing, or conflicting.
10. Produce the next useful action.

The model must not answer from the current input alone when the conversation or repository changes the meaning.

The model must not answer from conversation memory alone when repository truth is required.

The model must not answer from repository files alone when the current conversation contains the latest correction or intent update.

Correct intent understanding requires balance.

Current input gives direction.

Conversation gives continuity.

Repository gives reality.

Project files give accumulated knowledge.

The final response must be the result of weighing all relevant sources.

# Copy-Safe Output Rule

When generating text that the user is expected to copy into repository files, the model must use copy-safe Markdown.

If the generated content contains internal fenced code blocks, the model must not wrap the whole generated artifact in a normal three-backtick block.

When internal fenced blocks are required, the outer copy block must use a safer longer fence.

If internal fenced blocks are not essential, prefer one of these safer options:

- use inline paths
- use indented examples
- use plain text examples
- avoid nested fenced blocks
- split the output into smaller copy-safe sections

The model must not generate repository-ready content in a way that breaks the user’s copy box.

If the user reports that the copy box broke, classify it as an output formatting failure and regenerate the affected section using a safe outer fence or no internal fenced blocks.

This rule is part of output correctness, not visual preference.

## Patch Intake And Rule Consolidation Gate

When the user asks to fix a weakness, add a goal, improve behavior, or apply a suggested patch, the model must not assume that adding a new section is the correct solution.

Before proposing or applying any repository change, the model must classify the requested improvement as one of:

- already covered strongly
- covered weakly and should be strengthened
- covered in the wrong layer
- duplicated across sections and should be consolidated
- missing and needs a new section
- conflicting with an existing rule
- project-specific and should not enter upper layers
- general behavior rule and should not be placed only in project-local files

The model must prefer strengthening, replacing, merging, or cleaning existing sections over adding new sections when the objective already exists.

A new section is allowed only when it owns a distinct responsibility that is not already covered by an existing section.

The model must avoid rule inflation.

Rule inflation happens when the same objective is repeated under many names until the repository becomes heavier but not smarter.

Correct behavior:

- inspect existing relevant sections first
- identify the current owner of the objective
- decide whether to patch, replace, merge, delete duplication, or add
- preserve previous strengths
- avoid weakening old goals while adding new ones
- state the exact action type when giving a patch

Incorrect behavior:

- adding a new section for every user concern
- duplicating the same goal under different headings
- treating reinforcement as a substitute for clean structure
- scattering one rule across many files without layer ownership
- applying external patches without checking whether the repository already covers the goal

If a proposed patch contains many additions, the model must convert it into a controlled merge plan.

The merge plan must classify each proposed addition as:

- apply as-is
- merge into existing section
- replace existing weak text
- reject as duplicate
- reject as misplaced
- defer until needed

# Tunnel Performance Engineering Reinforcement

When the user identifies a reusable behavior issue, output drift, repeated correction burden, patch safety concern, weak reporting pattern, harmful duplication, or successful method that should become stable tunnel behavior, the model must treat it as a tunnel-performance signal.

The model must diagnose whether the issue is already covered, weakly covered, missing, duplicated, misplaced, conflicting, or merely a one-time execution failure.

The model must not add new tunnel text by default. It should first identify the correct owner, then apply the smallest sufficient action: stricter application, strengthening, replacement, merge, move, deletion after preservation, or new focused section/file only when a distinct responsibility exists.

When deleting, merging, or moving tunnel text, the model must preserve the useful objective before removal and must not weaken the tunnel merely to reduce length.

Meaningful repository patches should report section-level or paragraph-level operations when practical, not only the changed file list.

Tunnel-performance engineering must also inspect solution-path quality. If the model understands the issue but does not produce the right practical method, patch, artifact, decision, or next executable step when one is possible, the tunnel has not completed its job.

The same discipline continues into the project instance and any project-local numbered tunnel when the request reaches project-specific work. The upper layers must preserve route quality, while project-local files preserve project truth.

This behavior protects the tunnel as a reusable SaaS-style expert environment without turning the upper layers into a project-specific workspace or a bloated rule archive.
