# Mind AI Project Engineer

## v6.12 Central Operating Core

This package includes a one-touch central operating entry: `python .engineer/commands/engineer.py plug "<request>"`.
The plug output is not a passive report. It is the active operating context: route, protocol stack, tool keyboard, matrix, evidence gates, hard blocks, and receipt.



This repository equips a language model to work as a disciplined project engineer over whatever project is placed in `02_WORKSHOP/ACTIVE_PROJECT/`, while keeping the engineering runtime silent in normal project conversation.

The operational environment is silent. Its purpose is to provide navigation, project touching, reading, governance, and validation tools. The active project remains the center of conversation and work.

For AI systems and coding agents: read `AGENTS.md` first.

For users: place the real project inside `02_WORKSHOP/ACTIVE_PROJECT/`, then ask normally. The engineer should infer whether the project is new, documented, messy, mature, software, research, writing, business, or mixed.


## Project neutrality

No project-specific examples are embedded in the engineering runtime. The runtime is project-neutral by design. A distribution may ship either as a `template` with an empty `ACTIVE_PROJECT`, or as a `working_snapshot` with a real project already placed there. Project-specific identity, domain, scenario, module, technology stack, or execution priority is allowed only inside `02_WORKSHOP/ACTIVE_PROJECT/` and volatile generated outputs, and must still be discovered from current files.


## Surface truth lock

Project-facing answers should not score or explain the runtime. They must answer from active project evidence. If the active project has no meaningful material, the correct state is project formation: the engineer helps capture the first project truth instead of evaluating the environment.
