# ENTRY MAP

# Purpose

This file is the root entry map for the repository.

It tells any AI model, coding assistant, documentation assistant, or human collaborator how to enter this environment correctly before answering, generating, editing, reviewing, or reasoning about the project.

This file is not the project itself.

This file is not a project specification.

This file is not a replacement for the cognitive runtime, project orchestration, or project instance layers.

Its purpose is to prevent the model from starting in the wrong place, treating the layer files as the project, or answering without entering the required numbered tunnel.

## The Balance Scale Operating Loop

Before any meaningful context-dependent answer, the model must silently execute this loop:

1. Read the current user input.
2. Identify the real intent, not only the literal words.
3. Compare the input with the current conversation and latest user corrections.
4. Determine whether the request is new, cumulative, corrective, conflicting, continuation, audit, generation, repair, or exploration.
5. Enter the numbered repository tunnel from this file.
6. Pass through the cognitive runtime layer.
7. Pass through the project orchestration layer when project reasoning, project building, project organization, or project evolution is involved.
8. Enter the project instance layer.
9. Discover the active project dynamically.
10. Enter the project-local numbered tunnel when a project exists.
11. Inspect the relevant project files, parent files, child files, standards, taxonomies, or artifacts required by the request.
12. Compare the interpreted intent against repository reality.
13. Detect missing evidence, stale assumptions, contradictions, wrong layer placement, missing standards, unsupported claims, or placeholder artifacts.
14. Choose the correct action mode.
15. Answer, generate, patch, ask one focused question, inspect deeper, classify uncertainty, or propose a controlled improvement.

The model must not skip from user input directly to answer when repository context matters.

The model must not skip from user input directly to file generation when project standards or current artifact state matter.

The model must not skip from user input directly to general knowledge when project truth exists.

## Balance Scale Meaning Clarification

The balance scale is not a rule about making the visible answer symmetrical, neutral, or equally weighted.

It is a cognitive routing rule.

It means the model must weigh, in this order, the latest user input, relevant prior conversation threads, latest accepted corrections, repository tunnel rules, project instance evidence, and domain knowledge before answering.

The latest user input is the active intent anchor.

Prior conversation is used to recover connected threads, accepted decisions, and corrections that explain the current input.

The repository is used to ground the answer in persistent structure when available.

The project instance layer is used to ground the answer in actual project truth when project content exists.

If the project instance is minimal or empty, the model must not feel lost. It should treat the conversation and upper-layer methods as the active working context, then recommend or generate project-local standards only when they would improve continuity.

## Action Obligation Rule

When the user asks to solve, repair, audit, improve, regenerate, patch, continue, or start work, the model must not stop at analysis.

The response must include a clear practical action unless blocked by missing critical information.

Valid actions include:

- exact patch
- exact replacement text
- exact file to update
- exact next artifact to generate
- exact finding and correction
- exact decision recommendation
- exact question needed to unblock
- exact classification of repository weakness
- exact next inspection path

If the model only explains the problem when the user asked to fix or continue, that is a behavior failure.

## Sufficiency Before Patch Rule

Before proposing any patch, repository expansion, new standard, new section, new file, or structural improvement, the model must first determine whether the requested goal is already covered by the current repository.

The model must classify the situation as one of:

- missing rule
- weak rule
- misplaced rule
- duplicated rule
- conflicting rule
- existing sufficient rule
- behavior failure despite sufficient rule
- optional improvement, not required change

If the current repository already covers the goal sufficiently, the model must not generate another broad patch.

The correct action may be:

- confirm that the repository already covers the goal
- point to the existing governing rule or layer
- identify that the issue was a response-mode failure, not a repository gap
- recommend using the existing rule more strictly
- propose only a minimal reinforcement if a real enforcement weakness remains

Action does not always mean adding text.

Sometimes the correct action is to preserve the current structure, avoid duplication, and proceed using the existing rule.

Generating new documentation for an already-covered goal is a repository bloat failure.

A patch is justified only when it improves enforcement, removes contradiction, fixes placement, closes a missing rule, or reduces future ambiguity without weakening existing behavior.

When a user identifies a valuable behavior pattern, successful method, recurring friction, repeated explanation burden, or future SaaS operating improvement, the model must treat it as a tunnel-improvement candidate.

Before adding anything, the model must decide whether the improvement is already covered, weakly covered, misplaced, or missing. If covered, the model should use the existing rule more strictly. If weak or missing, the model should strengthen the smallest correct existing rule first before creating a new section or file.

The goal is cumulative tunnel intelligence without documentation bloat.

## Anti-Guessing Rule

The model must not claim that a file, folder, section, state, event, primitive, route, standard, or decision exists unless it has current repository evidence or explicit current user-provided evidence.

If the model cannot verify existence, it must say one of:

- not found in inspected evidence
- inferred from structure
- proposed addition
- likely candidate
- open question
- requires repository inspection

A guessed file name or guessed section title is not acceptable as project truth.

If the model accidentally guesses, the correct recovery is not only apology.

The correct recovery is:

1. classify the mistake as an evidence failure
2. check whether repository rules failed to prevent it
3. inspect the relevant tunnel or file
4. propose or apply the layer fix that prevents recurrence
5. continue with the corrected action

## Failure Detection And Self-Repair Rule

If the user reports that the model:

- guessed
- skipped the repository
- ignored the tunnel
- answered generically
- gave analysis without action
- contradicted accepted decisions
- forgot prior context
- changed terminology
- invented a path
- treated placeholder as mature
- generated inconsistent file content
- failed to infer a repository weakness from its own failure
- forced the user to repeatedly explain the same desired behavior
- solved the current task but revealed a reusable tunnel weakness
- demonstrated a useful new working method that should become stable repository behavior

then the model must treat this as a system behavior signal, not only as a local conversation mistake.

The model must ask internally:

- Which layer should have prevented this?
- Was the rule missing, weak, misplaced, duplicated, or not operational?
- Should the fix belong to cognitive runtime, project orchestration, project instance, project-local standards, or the target artifact?
- Is this a one-time correction, a reusable protocol gap, or a valuable operating pattern that should be preserved?
- Can the existing rule be tightened instead of adding a duplicate rule?

The model must then provide a repair path or confirm that the existing tunnel is sufficient and should simply be applied more strictly.

    -> 01_PROJECT_ORCHESTRATION/

# Hybrid Tunnel Runtime Rule

The tunnel is a unified operating path, not a collection of isolated reference files.

Markdown files define durable reasoning standards, contracts, boundaries, and detailed behavior.

YAML files define compact execution commands, activation gates, checklists, routing matrices, and validation triggers.

A YAML file does not replace the tunnel and does not override the numbered standards by itself.

It makes the tunnel more executable by binding a short command name to the relevant rules, standards, checks, and project-local contracts.

A preserved objective is not preserved because it was copied into a side folder. It is preserved when it is integrated into the active tunnel as one or more of:

- a numbered standard,
- a YAML command or gate,
- a checklist,
- a routing rule,
- a project-local contract,
- or a traceable implementation standard.

The model must treat command-style YAML as operational triggers. When a task matches a command trigger, the model must apply the referenced tunnel rules silently before answering, editing, generating, or reporting.

The model must also treat explicit measurable or structural user constraints as hard execution gates. If the user specifies a length, count, format, scope, language, citation, file, or no-unrelated-change constraint, the model must preserve and verify it before final output when possible.

If exact verification is unavailable, the model must produce a conservative output far enough below the limit and must not claim measured compliance without checking.
